服务器需要停止的服务：

NetworkManager

dnsmasq

在subnet里定义的网段要与你的网卡IP所处网段一致,

实际情况是网卡eth0没启动。

chkconfig --list dhcpd

centos 6 启动eth0 网卡时报错：

Error: Connection activation failed: Device not managed by NetworkManager

1\. Remove Network Manager from startup Services.

chkconfig NetworkManager off

chkconfig --list NetworkManager

2\. Add Default Net Manager

chkconfig network on

chkconfig --list network

Stop NetworkManager first

service NetworkManager stop

and then start Default Manager

service network start

or you can change it easy:

system-config-network 

uncheck the box that sets the eth0 as managed by NetworkManager and set IP info from there.

system-config-services 

stop and disable NetworkManager , start and enable network

you have problem because both of NetworkManager and Network using same Network driver and default network blocking access for NetworkManager. if you disable NetworkManager your problem will disappeared after restart.

squid Version 3.2.0.13: ERROR: No

forward-proxy ports configured.

&gt; #http/https ports

&gt; http_port 33070 transparent

&gt;

So, only one port, and its a NAT traffic receiving port. Looks like 

Squid was right, &quot;no forward-proxy ports configured&quot;.

Should not be on every request, that is a sign of some other problem 

happening. Just on the ones which need to produce internal URLs (ie 

error page icons, FTP directory icons, cache manager requests, etc). 

Squid requires a forward-proxy port for the URL it tells the browser to 

fetch those icons from. A NAT or TROXY receiving port is somewhat 

vulnerable to forgery attacks and Squid now avoids publishing its 

details. Accel ports have some problematic cases in the parser and until 

those are figured out properly they are avoided as well. Which just 

leaves the regular old forward-proxy port for now.

Adding &quot;http_port 3128&quot; or somesuch is enough to avoid this message
